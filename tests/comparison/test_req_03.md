# 需求文档：微服务架构迁移

## 1. 背景与目标

当前系统为单体应用（Java Spring Boot），随着业务增长，单体架构已暴露出部署耦合度高、扩展性差、故障隔离能力弱等问题。需将单体应用拆分为微服务架构：
- 按业务域拆分为用户服务、订单服务、商品服务、支付服务、通知服务等核心微服务
- 每个服务独立数据库（Database per Service），消除跨模块直接 SQL 访问

## 2. 模块拆分策略

- 采用领域驱动设计（DDD）方法识别限界上下文，以聚合根为拆分边界
- 拆分顺序：先拆通知服务（风险最低）→ 商品服务 → 订单服务 → 支付服务 → 用户服务
- 引入 API Gateway 统一入口，服务间通过 gRPC 同步通信 + Kafka 异步消息通信

## 3. 数据一致性保障

- 涉及跨服务事务的场景采用 Saga 模式（编排式），引入状态机管理补偿逻辑
- 最终一致性窗口需控制在 5 秒内，超出阈值触发人工干预告警
- 数据迁移采用 Change Data Capture（CDC）方案，通过 Debezium 监听源库 binlog 同步至新库

## 4. 风险点

- **服务间通信延迟**：原先进程内调用变为网络调用，延迟可能增加 10-50ms，需评估对核心链路的影响
- **分布式事务失败**：Saga 补偿逻辑覆盖不全可能导致数据不一致，需逐一梳理所有跨服务业务流程
- **运维复杂度上升**：服务数量从 1 个增至 5+，需配套建设服务注册、配置中心、链路追踪等基础设施
