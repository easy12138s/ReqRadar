# 需求文档：支付模块安全重构

## 1. 背景与目标

现有支付模块采用明文存储用户银行卡号、CVV 等敏感信息，存在严重的安全合规风险。需对支付模块进行全面安全重构：
- **加密存储改造**：将所有敏感支付信息（卡号、CVV、有效期）改为 AES-256-GCM 加密存储，密钥由 KMS 统一管理
- **支付网关抽象层**：引入统一的支付网关接口（PaymentGateway），将支付宝、微信支付、银联等第三方支付的对接逻辑从业务代码中解耦

## 2. 数据迁移方案

- 存量明文数据需在运行时逐条加密迁移，预计涉及 50 万条记录
- 迁移采用双写策略：新数据写入加密字段，后台任务逐步加密历史数据
- 迁移完成后切换读取路径，废弃明文字段

## 3. 风险点

- **数据丢失风险**：迁移过程中如发生异常，可能导致部分记录加密失败但明文已清除
- **第三方接口变更风险**：支付网关抽象层上线后，若第三方支付 SDK 升级，需同步更新适配器

## 4. 约束条件

- **不能停服迁移**：支付服务 7×24 可用，不允许计划内停机窗口
- **必须向后兼容**：旧版本客户端（含 App 和 H5）在重构上线后 3 个月内仍需正常完成支付流程
