## 一、项目定位

### 1.1 愿景

ReqRadar（需求透视）是一个**固定流程的垂直领域 Agent**，服务于需求从提出到开发之间的信息对齐环节。通过聚合散落在需求文档、代码仓库、历史记录中的关键信息，在关键节点为团队提供共用的上下文参考。

### 1.2 核心特征

| 特征 | 说明 |
|:---|:---|
| **固定流程** | 每次分析严格按照预定 6 步顺序执行，不根据中间结果动态调整路径 |
| **固定模板** | 输出报告的结构、章节、字段完全预定义，LLM 仅填充自然语言内容 |
| **确定性优先** | 相同输入在任何时间执行，报告结构和关键结论一致，行为可预期 |
| **决策导向** | 报告采用双层结构（决策摘要 + 技术支撑），每个结论附证据，直接支撑评审决策 |
| **隐私优先** | 敏感数据本地处理，仅输出脱敏摘要，原始内容不外传 |
| **记忆驱动** | 自动积累项目知识（术语、模块、贡献者），后续分析可复用历史上下文 |
| **工具增强** | LLM 可主动调用分析工具，多轮迭代获取充分信息 |

### 1.3 目标用户与价值

| 角色 | 价值 |
|:---|:---|
| **产品经理** | 提需求时预判技术影响面，降低沟通成本 |
| **部门领导/评审者** | 获取决策摘要与证据链，辅助评审决策 |
| **开发者** | 提前获得代码上下文、维护者信息和隐性约束，减少理解偏差 |

### 1.4 不做的事

- 不生成代码
- 不做实时监控
- 不提供自由对话式交互
- 不将原始敏感数据外传

## 二、双层报告设计

报告采用双层结构，服务不同角色的信息需求：

**第一层：决策摘要**（面向产品/管理层/评审人）
- 总体判断：是否建议推进及关键前提
- 结论与证据：决策主题、建议决策、依据、后续影响
- 支撑证据：证据类型、来源、摘要、可信度
- 待定问题与后续跟进

**第二层：技术支撑**（面向开发者/架构师）
- 需求理解、术语表、约束条件
- 影响域推断、代码命中、变更评估
- 风险分析、验证要点
- 建议评审人、实施建议

**附录**：流程完成度 / 内容完整度 / 证据支撑度三维度质量指标 + 项目知识上下文

设计原则：结论必有证据，影响范围从"模块计数"升级为"代码命中 + 推断影响域"，质量指标反映实际内容质量而非流程过程。

## 三、工具调用循环

Step 5（analyze）支持 LLM 通过多轮工具调用获取信息：

```
LLM 调用工具 → 获取结果 → LLM 继续分析 → 调用更多工具 → ... → 输出最终结构化结果
```

**9 种分析工具**：search_code、search_requirements、list_modules、get_contributors、read_file、read_module_summary、get_project_profile、get_terminology、get_dependencies

**安全机制**：ToolCallTracker 管理去重、轮次上限（默认 15 轮）和 token 预算（默认 8000）。

## 四、固定执行流程

| 步骤 | 名称 | 动作 | 输出 |
|:---:|:---|:---|:---|
| 1 | read | 读取需求文档纯文本 | 文本内容 |
| 2 | extract | 调用 LLM 提取关键词和业务术语 | 术语列表 + 摘要 |
| 3 | map_keywords | 将中文业务术语映射为英文代码搜索词 | 扩展关键词列表 |
| 4 | retrieve | 检索相似历史需求 + 智能匹配代码模块 | 相似需求 + 影响模块 |
| 5 | analyze | 工具调用循环 + 深度分析 + 生成决策摘要和证据链 | DeepAnalysis + DecisionSummary + EvidenceItems |
| 6 | generate | 组织双层报告内容（不重新判断风险和范围） | 双层 Markdown 报告 |

**降级策略**：

| 步骤 | 失败时降级行为 |
|:---|:---|
| Step 2: 术语提取 | 使用 TF-IDF 规则提取关键词 |
| Step 3: 关键词映射 | 使用原始术语作为搜索词 |
| Step 4: 智能匹配 | 降级到基于关键词的符号匹配 |
| Step 5: 深度分析 | 返回空列表，报告标注 |
| Step 6: 报告生成 | 使用模板占位文本填充 |

## 五、项目记忆系统

每次分析后自动更新：术语积累、团队信息记录、分析历史、模块关联历史。记忆存储在 `.reqradar/memory/memory.yaml`，后续分析自动注入 LLM prompt。

## 七、常见问题

### Q: 为什么选择固定流程而不是动态 Agent？
A: 需求分析是一个结构化任务，核心价值在于提供一致、可预期的报告。动态 Agent 行为不可预测，难以保证报告质量。

### Q: 双层报告解决了什么问题？
A: 之前的报告面向单一角色，产品经理看不懂技术细节，开发者觉得结论缺乏依据。双层设计让每个角色直接看到自己需要的信息，且结论必附证据。
